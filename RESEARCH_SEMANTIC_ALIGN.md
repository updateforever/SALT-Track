# SALT-Track 研究文档

## 1. 项目定位

SALT-Track 不是一个从零开始重写的视觉语言跟踪器，而是一个基于 ATCTrack 持续推进的研究型项目。

我们当前的研究目标很明确：

**探索文本在视觉语言跟踪中除了“输入条件”之外，是否还能作为训练阶段的监督信号，进一步约束模型学到更稳定的目标语义表示。**

这份文档服务于项目推进本身，因此允许保留真实的科研思考过程，不强求与最终论文写作故事完全一致。

## 2. 研究起点

我们最初的思路来自一篇相关工作的启发。那篇文章把“语义对齐”作为一个有效的监督思路来使用，这给了我们一个直接问题：

- 在视觉语言跟踪里，是否也可以加入某种语义对齐损失？

沿着这个方向继续分析后，我们发现真正值得研究的问题并不是“再额外引入一个外部语义空间”，而是：

- ATCTrack 本身已经有较强的视觉-语言交互能力
- 文本侧也已经能够输出增强后的目标相关语义表示
- 如果目标是研究文本监督，那么更自然的做法应该是直接在 ATCTrack 自身特征体系内构造监督，而不是再强行接一个外部语义空间

因此，项目路线发生了一个关键收缩：

- 保留“语义对齐”这个大方向
- 但放弃已经不适用的外部 CLIP 对齐设想
- 转而研究 ATCTrack 内部的实例特征与文本特征之间的监督关系

## 3. 当前核心问题

我们当前聚焦的问题可以写成一句话：

**跟踪器不仅要知道框住了哪里，还要知道框住的是不是文本描述中的那个目标。**

这其实对应两类不同性质的监督：

### 3.1 感知监督信号

这是传统跟踪任务天然具备的监督，包括：

- GIoU
- L1
- 置信度
- 框回归相关目标

这一类监督回答的是：

- 预测框的位置是否准确
- 模型是否在视觉层面感知到了目标

### 3.2 语义认知监督信号

这是我们当前新增的研究重点。

它不再只问：

- 框是不是落在一个“像目标”的区域上

而是进一步问：

- 这个区域是否真的符合文本描述的目标语义

因此我们更愿意把这条辅助监督理解为：

- 一种实例级的语义认知监督信号

## 4. 方法视角的转变

项目目前最大的视角变化，是把“创新点”从输入端转向监督端。

原始 ATCTrack 已经做了很多输入端工作：

- 动态目标-上下文建模
- 文本目标词选择
- 视觉语言融合
- 时序状态建模

如果继续沿着“怎么更好地使用文本输入”去讲，很容易陷入和原方法边界不清的问题。

但如果换一个角度，问题会更清楚：

- 原始框监督主要是感知层面的
- 我们补充的是语义认知层面的监督
- 这样方法的定位就从“重新设计文本输入”转向“补充一条新的监督信号”

这也是当前最适合后续论文写作和图示表达的方向。

## 5. 当前最小实现

目前仓库中的实现已经收敛到一个最小可验证版本。

### 5.1 视觉侧

在 ATCTrack 融合后的搜索特征图上：

- 用预测框提取预测实例特征
- 用 GT 框提取 GT 实例特征

两者都来自同一个跟踪特征空间。

### 5.2 文本侧

文本不再接一个额外的外部编码器分支。

当前直接使用：

- ATCTrack 文本分支增强后的文本特征
- 再通过 masked mean pooling 得到句子级文本语义锚点

这个语义锚点代表的是：

- 跟踪网络当前真正用来理解目标描述的文本表示

### 5.3 监督侧

当前语义辅助监督由两部分构成：

- 预测实例特征与 GT 实例特征之间的视觉一致性约束
- 预测实例特征与文本语义锚点之间的语义一致性约束

同时我们引入了一个动态门控思想：

- 如果 GT 特征比 Pred 特征更接近文本，说明文本对当前样本具有纠偏价值
- 如果 GT 自身都不再更接近文本，说明这条文本监督在当前样本上不够可靠，应当减弱

因此，文本监督不是固定强加，而是被设计成一种带可靠性估计的辅助监督。

## 6. 为什么不再走 CLIP 路线

这一点需要明确写死，避免后续文档和代码再次分叉。

### 6.1 理由一：研究目标已经改变

我们要研究的是：

- 文本如何作为监督信号帮助 ATCTrack 训练

不是：

- 如何把跟踪器特征对齐到另一个外部语义空间

### 6.2 理由二：ATCTrack 内部已经有可用文本表示

当前我们已经能拿到：

- ATCTrack 增强后的文本特征
- 预测区域与 GT 区域的实例级视觉特征

这三者已经足以构造一个闭环监督。

### 6.3 理由三：额外引入外部对齐空间会让问题变得更混乱

如果继续引入 CLIP：

- 会把研究重点从“文本监督”转向“跨空间对齐”
- 会增加额外模块与工程复杂度
- 也会削弱当前方法的最小性和可解释性

因此，当前项目不再保留 CLIP 作为主线方案。

## 7. 当前实验组织方式

为了让后续结果更清楚，我们目前把正式实验组织为四类：

1. `atctrack_base_semantic_full`
   - 主实验
   - 视觉一致性监督 + 文本语义监督
   - 全量微调

2. `atctrack_base_semantic_visual`
   - 只保留视觉一致性监督
   - 用于判断语义文本监督是否真的带来额外收益

3. `atctrack_base_semantic_text`
   - 只保留文本语义监督
   - 用于判断文本监督单独是否成立

4. `atctrack_base_semantic_lora`
   - 使用 LoRA 微调的训练方式对比
   - 主要用于比较训练范式，而不是当前主线方法本身

## 8. 当前训练策略

当前正式训练配置采用以下统一策略：

- batch size：25
- epoch：100
- lr：8e-5
- lr drop epoch：80
- save interval：10
- 语义监督采用两阶段调度：
  - 前 40% epoch 使用 0.3
  - 后 60% epoch 使用 0.1

这个设计背后的考虑是：

- 前期让语义监督有足够存在感
- 后期让它继续保留作用，但不过度压制定位主任务

## 9. 当前工程入口

### 9.1 正式训练入口

统一使用：

- `lib/train/run_training.py`

不再建议使用旧的：

- `tracking/train.py`

原因是后者只是一个旧包装器，而且会额外引入多卡和环境变量层面的混淆。

### 9.2 训练、推理与分析命令

统一查看：

- `scripts/cmd.md`

这份命令文档目前是项目运行的唯一有效整理入口。

## 10. 当前已有判断

截至目前，我们已经形成几个比较稳定的中间判断：

1. 文本监督方向是值得继续做的
2. 语义监督更适合被写成“监督信号创新”，而不是“输入模块创新”
3. 当前工作最适合用“感知监督 + 语义认知监督”的二元结构来表述
4. 语义监督不能过强，也不能弱到没有存在感，因此需要阶段调度
5. 外部 CLIP 路线不再适合当前项目主线

## 11. 当前仍需继续验证的问题

后续仍需要围绕以下问题继续做实验：

### 11.1 文本监督到底带来多少真实收益

需要通过：

- `full`
- `visual`
- `text`

三组结果对比来回答。

### 11.2 语义权重调度是否合理

目前采用：

- 前 40%：0.3
- 后 60%：0.1

这是一个工程上合理的起点，但还不是最终结论。

### 11.3 不同数据集组合是否会影响语义监督效果

尤其是：

- 纯语言跟踪数据集
- 混合语言质量差异的数据集
- VastTrack / OTB99 / TNL2K / LaSOT 的组合比例

## 12. 对后续论文写作的提醒

当前项目文档允许保留真实科研思考过程，但后续论文写作不一定要原样照搬。

目前更适合论文故事线的表达方式是：

- 传统 VLT 主要依赖感知型监督
- 仅靠框级监督不足以约束“语义正确性”
- 因此我们引入语义认知监督信号
- 让模型不仅学会“看见目标”，还学会“确认目标是否符合文本语义”

这比直接讲“我们想试一下语义对齐”更收敛，也更适合方法图和摘要表述。
